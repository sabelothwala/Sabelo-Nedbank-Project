# Output Schema Specification — DE Challenge Gold Layer

**Document ID:** D-10
**Spec references:** `de_challenge_design_spec.md` §4.2; `de_data_requirements_spec.md` §6 (GAP-026); `de_build_execution_plan.md` §D-10
**Status:** Final

---

## 1. Overview

All pipeline output must conform to the following schema and format requirements.

**Format:** Delta Parquet (all layers)

**Output root:**
- Batch tables: `/data/output/gold/`
- Stage 3 streaming tables: `/data/output/stream_gold/`

**Compatibility:** All Gold tables must be readable by both DuckDB and PySpark without schema transformation.

**Surrogate key generation:** Surrogate keys (`_sk` fields) must be unique, non-null, and stable across pipeline re-runs on the same input data. Acceptable approaches:
- `row_number()` with a stable `ORDER BY` on the natural key
- `sha2(natural_key, 256)` cast to BIGINT (deterministic hash)

## 2. `fact_transactions` (15 fields)

| # | Field | Type | Nullable | Source Derivation |
|---|---|---|---|---|
| 1 | `transaction_sk` | BIGINT | No | Surrogate key. Generated by pipeline. Unique, non-null. Use deterministic hash or `row_number()` with stable `ORDER BY transaction_id`. |
| 2 | `transaction_id` | STRING | No | Natural key. From `transactions.jsonl`. Deduplicated in Silver layer — one row per unique `transaction_id`. |
| 3 | `account_sk` | BIGINT | No | Foreign key to `dim_accounts.account_sk`. Resolved via `transactions.account_id` → `dim_accounts.account_id`. |
| 4 | `customer_sk` | BIGINT | No | Foreign key to `dim_customers.customer_sk`. Resolved via `account_sk` → `dim_accounts.customer_id` → `dim_customers.customer_id`. |
| 5 | `transaction_date` | DATE | No | From `transactions.jsonl`. Standardised to `DATE` type (ISO format `YYYY-MM-DD`). |
| 6 | `transaction_timestamp` | TIMESTAMP | No | From `transactions.jsonl`. Combined `transaction_date` + `transaction_time`. Standardised to TIMESTAMP (UTC-aware where possible). |
| 7 | `transaction_type` | STRING | No | From `transactions.jsonl`. Values: `DEBIT`, `CREDIT`, `FEE`, `REVERSAL`. |
| 8 | `merchant_category` | STRING | Yes | From `transactions.jsonl`. 20 MCC-style codes (e.g., `GROCERY`, `FUEL`, `RESTAURANT`). Nullable where absent in source. |
| 9 | `merchant_subcategory` | STRING | Yes | From `transactions.jsonl`. **Null in Stage 1 output** (field absent from Stage 1 source data). Present in Stage 2 and Stage 3. Pipeline must not fail if field is absent. |
| 10 | `amount` | DECIMAL(18,2) | No | From `transactions.jsonl`. Must be numeric after type standardisation. Two decimal places. |
| 11 | `currency` | STRING | No | From `transactions.jsonl`. **Must be standardised to `"ZAR"`** regardless of source value variants. |
| 12 | `channel` | STRING | No | From `transactions.jsonl`. Values: `POS`, `APP`, `ATM`, `EFT`, `USSD`, `INTERNAL`. |
| 13 | `province` | STRING | Yes | From `transactions.jsonl` (`location.province`). Must match the province of the customer linked to the account. |
| 14 | `dq_flag` | STRING | Yes | Pipeline-derived. **Null if record is clean.** Set to one of the DQ issue codes if flagged (see Section 8 — Validation Notes). |
| 15 | `ingestion_timestamp` | TIMESTAMP | No | Pipeline-generated. Timestamp when the record entered the Bronze layer. Must be consistent with Bronze ingestion run time. |

## 3. `dim_accounts` (11 fields)

| # | Field | Type | Nullable | Source Derivation |
|---|---|---|---|---|
| 1 | `account_sk` | BIGINT | No | Surrogate key. Generated by pipeline. Unique, non-null. |
| 2 | `account_id` | STRING | No | Natural key. From `accounts.csv`. |
| 3 | `customer_id` | STRING | No | **FROM `accounts.csv.customer_ref`.** Renamed to `customer_id` in the Gold layer for semantic clarity. Required for Validation Query 2 join: `dim_accounts.customer_id = dim_customers.customer_id`. |
| 4 | `account_type` | STRING | No | From `accounts.csv`. Values: `SAVINGS`, `TRANSACTIONAL`, `CREDIT`. |
| 5 | `account_status` | STRING | No | From `accounts.csv`. Values: `ACTIVE`, `DORMANT`, `CLOSED`, `SUSPENDED`. |
| 6 | `open_date` | DATE | No | From `accounts.csv`. Standardised to `DATE` type (`YYYY-MM-DD`). |
| 7 | `product_tier` | STRING | No | From `accounts.csv`. Values: `BASIC`, `STANDARD`, `PREMIUM`. |
| 8 | `digital_channel` | STRING | No | From `accounts.csv`. |
| 9 | `credit_limit` | DECIMAL(18,2) | Yes | From `accounts.csv`. Null for non-CREDIT account types. |
| 10 | `current_balance` | DECIMAL(18,2) | No | From `accounts.csv`. Snapshot balance at time of data generation. Two decimal places. |
| 11 | `last_activity_date` | DATE | Yes | From `accounts.csv`. Standardised to `DATE` type. May be null in Stage 2 (DQ-injected). |

**Pipeline derivation note:** The Bronze layer preserves `customer_ref` as-is from `accounts.csv`. The Silver layer validates referential integrity (every `customer_ref` maps to a valid `customer_id` in `customers.csv`). The Gold layer renames `customer_ref` → `customer_id` in `dim_accounts`.

## 4. `dim_customers` (9 fields)

| # | Field | Type | Nullable | Source Derivation |
|---|---|---|---|---|
| 1 | `customer_sk` | BIGINT | No | Surrogate key. Generated by pipeline. Unique, non-null. |
| 2 | `customer_id` | STRING | No | Natural key. From `customers.csv`. |
| 3 | `gender` | STRING | No | From `customers.csv`. Values: `M`, `F`, `NB`, `UNKNOWN`. |
| 4 | `province` | STRING | No | From `customers.csv`. One of 9 SA provinces. |
| 5 | `income_band` | STRING | No | From `customers.csv`. |
| 6 | `segment` | STRING | No | From `customers.csv`. |
| 7 | `risk_score` | INTEGER | No | From `customers.csv`. Range: 1–10. |
| 8 | `kyc_status` | STRING | No | From `customers.csv`. Values: `VERIFIED`, `PENDING`, `FAILED`. |
| 9 | `age_band` | STRING | No | **Derived from `customers.csv.dob`** (not copied directly). Compute as of the pipeline run date. Buckets: `"18-25"`, `"26-35"`, `"36-45"`, `"46-55"`, `"56-65"`, `"65+"`. |

**`age_band` derivation rule:**
```
age = floor((pipeline_run_date - dob) / 365.25)
age_band = CASE
  WHEN age >= 65 THEN "65+"
  WHEN age >= 56 THEN "56-65"
  WHEN age >= 46 THEN "46-55"
  WHEN age >= 36 THEN "36-45"
  WHEN age >= 26 THEN "26-35"
  WHEN age >= 18 THEN "18-25"
  ELSE NULL  -- should not occur given generation constraints
END
```

## 5. `current_balances` (Stage 3 only — 4 fields)

Written to `/data/output/stream_gold/current_balances/`.

Updated by the streaming pipeline as each micro-batch of events is processed. One row per `account_id` — this is an upsert/merge table, not an append table.

| # | Field | Type | Nullable | Description |
|---|---|---|---|---|
| 1 | `account_id` | STRING | No | Natural key. References a valid account from the batch pipeline. |
| 2 | `current_balance` | DECIMAL(18,2) | No | Most recent computed balance for this account, updated after each streaming event. |
| 3 | `last_transaction_timestamp` | TIMESTAMP | No | Timestamp of the most recent transaction event processed for this account. |
| 4 | `updated_at` | TIMESTAMP | No | When this row was last written by the streaming pipeline. Must be within 5 minutes of the source event timestamp (SLA). |

## 6. `recent_transactions` (Stage 3 only — 7 fields)

Written to `/data/output/stream_gold/recent_transactions/`.

Maintains the **last 50 transactions per account**. As new events arrive, older records beyond position 50 are evicted. Implemented as a Delta merge/upsert keyed on `(account_id, transaction_id)`.

| # | Field | Type | Nullable | Description |
|---|---|---|---|---|
| 1 | `account_id` | STRING | No | Account reference. |
| 2 | `transaction_id` | STRING | No | Unique transaction identifier from the streaming event. |
| 3 | `transaction_timestamp` | TIMESTAMP | No | Timestamp of the transaction event. |
| 4 | `amount` | DECIMAL(18,2) | No | Transaction amount. |
| 5 | `transaction_type` | STRING | No | Values: `DEBIT`, `CREDIT`, `FEE`, `REVERSAL`. |
| 6 | `channel` | STRING | Yes | Channel used. Nullable (may be absent from stream events). |
| 7 | `updated_at` | TIMESTAMP | No | When this row was written by the streaming pipeline. |

**Retention rule:** Only the 50 most recent transactions (by `transaction_timestamp`) per `account_id` are retained. Rows beyond position 50 must be deleted on each merge cycle.

## 7. Output Directory Structure

```
/data/output/
  bronze/
    accounts/           (Delta Parquet — raw accounts.csv + ingestion_timestamp)
    customers/          (Delta Parquet — raw customers.csv + ingestion_timestamp)
    transactions/       (Delta Parquet — raw transactions.jsonl + ingestion_timestamp)
  silver/
    accounts/           (Delta Parquet — typed, deduplicated)
    customers/          (Delta Parquet — typed, deduplicated)
    transactions/       (Delta Parquet — typed, deduplicated, DQ-flagged)
  gold/
    fact_transactions/  (Delta Parquet — 15 fields as per Section 2)
    dim_accounts/       (Delta Parquet — 11 fields as per Section 3, including customer_id)
    dim_customers/      (Delta Parquet — 9 fields as per Section 4)
  stream_gold/          (Stage 3 only)
    current_balances/   (Delta Parquet — 4 fields as per Section 5)
    recent_transactions/ (Delta Parquet — 7 fields as per Section 6)
  dq_report.json        (Stage 2 and Stage 3 — DQ summary report)
```

**Path conventions:**
- All paths are relative to the container's `/data/output/` mount point.
- Each table directory must contain valid Delta Lake metadata (`_delta_log/`) alongside the Parquet part files.
- The scoring harness reads tables using `delta.load("path/to/table")` (PySpark) or `delta_scan("path/to/table")` (DuckDB + `delta` extension).

## 8. Validation Notes

The scoring harness applies the following automated checks against Gold layer output.

### Surrogate Keys
- `transaction_sk`, `account_sk`, `customer_sk` must each be unique within their respective tables.
- No null values permitted in any surrogate key column.

### Foreign Key Integrity
- Every `fact_transactions.account_sk` must match a `dim_accounts.account_sk`.
- Every `fact_transactions.customer_sk` must match a `dim_customers.customer_sk`.
- Every `dim_accounts.customer_id` must match a `dim_customers.customer_id` (Validation Query 2).
- Zero orphaned fact rows are permitted.

### Currency Standardisation
- `fact_transactions.currency` must be `"ZAR"` for every row, regardless of source variants.

### Date Format Consistency
- All `DATE` columns must use ISO format (`YYYY-MM-DD`).
- All `TIMESTAMP` columns must use ISO 8601 format.
- Mixed formats within a column constitute a schema conformance failure.

### DQ Flag Values
The `dq_flag` column in `fact_transactions` must be either:
- `NULL` — record passed all quality checks (clean record), or
- One of the 6 recognised issue codes:
  - `ORPHANED_ACCOUNT` — transaction references an account not in `dim_accounts`
  - `DUPLICATE_DEDUPED` — record was identified as a duplicate and removed/flagged
  - `TYPE_MISMATCH` — field value does not match expected data type
  - `DATE_FORMAT` — date field was in an unrecognised or inconsistent format
  - `CURRENCY_VARIANT` — currency field contained a value other than `"ZAR"`
  - `NULL_REQUIRED` — a non-nullable field contained a null value

No values outside this enumeration are accepted.

### Schema Conformance (Scoring: 5 points)
The schema diff check verifies:
- `dim_accounts` has exactly 11 fields, with `customer_id` at position 3.
- `fact_transactions` has exactly 15 fields, with `merchant_subcategory` present and nullable.
- `dim_customers` has exactly 9 fields, with `age_band` derived (not a raw `dob` copy).
- Streaming tables present in Stage 3 output with correct field counts (4 and 7 respectively).

### Streaming SLA (Stage 3)
- `current_balances.updated_at` and `recent_transactions.updated_at` must be within 300 seconds (5 minutes) of the source event timestamp for full SLA credit.
- Latency between 300–600 seconds receives partial credit.
- Latency exceeding 600 seconds receives no SLA credit.
